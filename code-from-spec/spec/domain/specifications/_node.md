# ROOT/domain/specifications

## Intent

Defines the structure of specifications — nodes, logical
names, frontmatter format, parent and subject
relationships, and the spec comment format used in
generated files.

## Contracts

### Location

Specifications live under `code-from-spec/`.

### Nodes

Every spec node is a directory containing a `_node.md`
file. The directory structure is the spec tree — a node's
position in the filesystem is its position in the
hierarchy.

A node with child directories is an **intermediate node**.
A node without children is a **leaf node**. Only leaf
nodes may have `implements`.

A **test node** is a file ending in `.test.md` placed
inside the directory of the node it tests. The canonical
test node is named `default.test.md`. Additional test
nodes use `<name>.test.md`. Any node may have test nodes.

```
config/
  _node.md             ← spec node (leaf)
  default.test.md      ← canonical test node
  edge_cases.test.md   ← additional test node
```

### Logical names

Every node has a logical name derived from its position in
the tree. Spec nodes use the `ROOT/` prefix; test nodes
use the `TEST/` prefix.

| Logical name | Resolves to |
|---|---|
| `ROOT` | `code-from-spec/_node.md` |
| `ROOT/architecture/backend` | `code-from-spec/architecture/backend/_node.md` |
| `ROOT/architecture/backend/config` | `code-from-spec/architecture/backend/config/_node.md` |
| `TEST/architecture/backend/config` | `code-from-spec/architecture/backend/config/default.test.md` |
| `TEST/architecture/backend/config(edge_cases)` | `code-from-spec/architecture/backend/config/edge_cases.test.md` |

Resolution rules:
- `ROOT` → `code-from-spec/_node.md`
- `ROOT/<path>` → `code-from-spec/<path>/_node.md`
- `TEST/<path>` → `code-from-spec/<path>/default.test.md`
- `TEST/<path>(<name>)` → `code-from-spec/<path>/<name>.test.md`
- `TEST/<path>` is an alias for `TEST/<path>(default)`

A `depends_on` entry may include a subsection qualifier
in parentheses — e.g., `ROOT/x/y(interface)`. The
qualifier identifies a specific subsection of the
referenced node but does not change which file is
resolved. `PathFromLogicalName("ROOT/x/y(z)")` resolves
to the same file as `PathFromLogicalName("ROOT/x/y")`.

### Parent relationships

- The root node (`ROOT`) has no parent.
- A spec node's parent is determined by its logical name:
  `ROOT/x/y/z` has parent `ROOT/x/y`.
- Test nodes have no parent in the tree. They have a
  **subject**: the `_node.md` in the same directory.
  The subject of `TEST/x/y` and `TEST/x/y(name)` is
  `ROOT/x/y`.

### Frontmatter

Every node begins with a YAML frontmatter block. The
following fields are relevant for staleness verification:

| Field | Description | Notes |
|---|---|---|
| `version` | Current version of this node. | All nodes |
| `parent_version` | The version of the parent node this node was written against. | Spec nodes only (omitted by root) |
| `subject_version` | The version of the subject node this test was written against. | Test nodes only |
| `depends_on` | Cross-tree dependencies with their known versions. Uses logical names. | Optional |
| `implements` | Source files generated by this node. | Leaf and test nodes |

Other frontmatter fields may exist and must be ignored.

Example — root node:

```yaml
---
version: 3
---
```

Example — intermediate node:

```yaml
---
version: 2
parent_version: 3
---
```

Example — leaf node with dependencies:

```yaml
---
version: 1
parent_version: 1
depends_on:
  - path: ROOT/architecture/backend/celcoin-gateway
    version: 6
implements:
  - internal/configuration/config.go
---
```

Example — test node:

```yaml
---
version: 1
subject_version: 2
implements:
  - internal/configuration/config_test.go
---
```

### Spec comment format

Generated files carry a spec reference comment on the
first line where a comment is allowed:

```
<comment-prefix> code-from-spec: <logical-name>@v<version>
```

The comment syntax follows the language's single-line
comment convention. The embedded string is fixed.

Example — a spec node with `version: 5`:

```go
// code-from-spec: ROOT/architecture/backend/config@v5
package configuration
```

Example — a test node with `version: 3`:

```go
// code-from-spec: TEST/architecture/backend/config@v3
package configuration
```

`TEST/x` and `TEST/x(default)` are both valid forms in
spec comments — either must match.
