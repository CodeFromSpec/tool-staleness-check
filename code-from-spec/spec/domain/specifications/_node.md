# ROOT/domain/specifications

## Intent

Defines the structure of specifications — nodes, logical
names, frontmatter format, parent relationships, and the
spec comment format used in generated files.

## Contracts

### Location

Specifications live under `spec/`.

### Nodes

Every spec node is a directory containing a `_node.md`
file. The directory structure is the spec tree — a node's
position in the filesystem is its position in the
hierarchy.

A node with child directories is an **intermediate node**.
A node without children is a **leaf node**. Only leaf
nodes may have `implements`.

A **test node** is a file ending in `.test.md` placed
inside the directory of the leaf node it tests. The
canonical test node is named `default.test.md`. Additional
test nodes use `<name>.test.md`. Only leaf nodes (those
with `implements`) may have test nodes — test nodes
alongside intermediate nodes are invalid.

```
config/
  _node.md             ← spec node (leaf)
  default.test.md      ← canonical test node
  edge_cases.test.md   ← additional test node
```

### Logical names

Every node has a logical name derived from its position in
the tree. Spec nodes use the `ROOT/` prefix; test nodes
use the `TEST/` prefix.

| Logical name | Resolves to |
|---|---|
| `ROOT` | `spec/_node.md` |
| `ROOT/architecture/backend` | `spec/architecture/backend/_node.md` |
| `ROOT/architecture/backend/config` | `spec/architecture/backend/config/_node.md` |
| `TEST/architecture/backend/config` | `spec/architecture/backend/config/default.test.md` |
| `TEST/architecture/backend/config(edge_cases)` | `spec/architecture/backend/config/edge_cases.test.md` |

Resolution rules:
- `ROOT/x` → `spec/x/_node.md`
- `TEST/x(y)` → `spec/x/y.test.md`
- `TEST/x` is an alias for `TEST/x(default)`

### Parent relationships

- The root node (`ROOT`) has no parent.
- A spec node's parent is determined by its logical name:
  `ROOT/x/y/z` has parent `ROOT/x/y`.
- A test node's parent is the `_node.md` in the same
  directory: `TEST/x/y` and `TEST/x/y(name)` both have
  parent `ROOT/x/y`.

### Frontmatter

Every node begins with a YAML frontmatter block. The
following fields are relevant for staleness verification:

| Field | Description | Notes |
|---|---|---|
| `version` | Current version of this node. | All nodes |
| `parent_version` | The version of the parent node this node was written against. | Omitted by the root node |
| `depends_on` | Cross-tree dependencies with their known versions. Uses logical names. | Optional |
| `implements` | Source files generated by this node. | Leaf and test nodes |

Other frontmatter fields may exist and must be ignored.

Example — root node:

```yaml
---
version: 3
---
```

Example — intermediate node:

```yaml
---
version: 2
parent_version: 3
---
```

Example — leaf node with dependencies:

```yaml
---
version: 1
parent_version: 1
depends_on:
  - path: EXTERNAL/celcoin-api
    version: 5
  - path: ROOT/architecture/backend/celcoin-gateway
    version: 6
implements:
  - internal/configuration/config.go
---
```

Example — test node:

```yaml
---
version: 1
parent_version: 2
implements:
  - internal/configuration/config_test.go
---
```

### Spec comment format

Generated files carry a spec reference comment on the
first line where a comment is allowed:

```
<comment-prefix> spec: <logical-name>@v<version>
```

The comment syntax follows the language's single-line
comment convention (`//` for Go/Java/C, `#` for
Python/Ruby, etc.).

Logical name derivation from node path:
- Spec node `spec/x/_node.md` → `ROOT/x`
- Test node `spec/x/default.test.md` → `TEST/x`
- Test node `spec/x/name.test.md` → `TEST/x(name)`

`TEST/x` and `TEST/x(default)` are both valid forms in
spec comments — either must match.

Example — a spec node titled
`ROOT/architecture/backend/config` with `version: 5`:

```go
// spec: ROOT/architecture/backend/config@v5
package configuration
```

Example — a test node titled
`TEST/architecture/backend/config` with `version: 3`:

```go
// spec: TEST/architecture/backend/config@v3
package configuration
```
